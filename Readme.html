Oracle BI Mobile App Designer (BIMAD) plugin SDK

About

This SDK contains everything you need to start developing your own Oracle BI Mobile App Designer plugins. It consists of all the plugin javascript files, a development server as well as (sample) data. This completely self-contained SDK allows you to develop plugins without needing a full blown OBI production environment. You can develop and test everything on your local machine using this SDK and only deploy when you are ready.

Prerequisite

To work with this SDK, you need to install the following tools.

note that you might need root/admin privileges to install some of the software below

Node.js

Please install v0.10.x or higher.

Download and Install node.js

Grunt

Run the following command to install grunt:

$ npm install -g grunt-cli

If you are behind a firewall, you need to set a http proxy server.

on Linux, or Mac:

$ https_proxy=http://<your https proxy server>:<proxy port>/ npm install -g  grunt-cli

on Windows:

c:\> set HTTPS_PROXY=http://<https proxy server>:<proxy port>/
c:\> npm install -g grunt-cli

Setup your local environment

Once you have downloaded the SDK zip file from your Oracle BI Mobile App Designer, you need to unzip it and change your directory to the unzipped folder. In this folder you need to run following command:

$ npm install

Again, you might need to set your proxy server to be able to run this command. Try next command:

on Linux or Mac

$ https_proxy=http://<your https proxy server>:<proxy port>/ npm install
(ex $ https_proxy=https://proxy.xyz.com:8888/ npm install)

on Windows

c:\bimadplugin> set HTTPS_PROXY=http://<https proxy server>:<proxy port>/
c:\bimadplugin> npm install

This command will install all the node.js modules the SDK is dependant on (described in the package.json file). You should now have a fully working environment for local plugin development.

Files

Plugin file (.js)

The plugin file is located in the root directory of the SDK (after you unzip the downloaded SDK). It is in this file that you will code your plugin. It is the one and only javascript file that gets executed by the BIMAD plugin framework to run your plugin.

You can organize your plugin code in as many files as you like, just be aware that it is the Plugin file that gets called by the BIMAD framework, consider it your public static void main (java), the entry point to your code. Also note that this entry point can be changed (see next - Gruntfile.js)

Open this file in your favorite text editor or IDE to write your plugin code. Once you start the development server, any changes to your code will be immediately applied (after you SAVE your changes) and visible in the browser (courtesy of LiveReload).

Gruntfile.js

The Gruntfile.js controls the dev server. If you want to rename your plugin or if you want to start with a different JavaScript file (including from another directory), you need to change the value of the pluginFile on line 3.

Data folder (data/data01.csv)

The data folder contains the Sample data you picked when you downloaded the SDK, in CSV format. You can add your own CSV files containing different (sample) data, e.g. you can copy the data01.csv file to data02.csv file, open it up and change some values. You automatically have access to these different files (i.e. different data) in the plugin testing page. When you pick one, the plugin test page will refresh itself with the new data.

Each data file must contain the exact format of data as your plugin requires. I.e. if the plugin definition contains 2 field entries, the csv file must contain 2 columns and the order of those columns must be same as the field entries.

This SDK server does not provide any sorting, reordering or regrouping at the time of serving data.

This is just sample data for you to see the plugin working in your local environment with actual (albeit sample) data. Once you deploy your plugin to BIMAD, it gets access to the full array of data sources supported by BIMAD (database, semantic layer, Excel file ...)

node_modules, package_json

These files are required to maintain the SDK environment. Don't delete or edit them.

Trying sample plugin

If you already have sample plugin file to start with, you can replace the existing sample plugin with your own by following these steps:

First, put plugin javascript file into the bimadplugin folder. If you have xmp file, use unzip tool to extract javascript file in there. Also don't forget extract and copy asset files as well.

% cd ~/Downloads
% unzip com.oracle.YourPlugin.xmp YourPlugin.js
% cp YourPlugin.js bimadplugin/YourPlugin.js

Then open the copied plugin file and change the first 4 lines (area surrounded with **) to match your new plugin definition. 

{
   id: '**com.oracle.yourpluginid**',
   component: {
     'name': '**YourPlugin**',
     'tooltip': 'Insert **YourPlugin**'
   },

Finally, open Gruntfile.js and change the the value of pluginFile variable to the same file name as your plugin file.

var pluginFile = 'YourPlugin.js'

Hosted files

If your plugin needs other assets such as images, style sheets or other JavaScript files, these can be placed in the assets directory. They can be referenced using the following 2 techniques in the plugin code:

Note: jQuery 1.11 and jQuery Mobile 1.4.2 are loaded with Mobile App. You don't need to load them by yourself.

http protocol

io/viewer/get-plugin-asset/<your plugin id> will give you access to the resource under assets folder.

io/viewer/get-plugin-asset/<your plugin id>/<file path>

E.g. the following code snippet will show your logo.png image that you placed in assets/images:

render: function(context, containerElem, data, fields, props) {
    containerElem.innerHTML = '<img src="io/viewer/get-plugin-asset/    com.oracle.xdo.Sample/images/logo.png"/>';
}

asset protocol

Since the regular http protocol notation is long, we created a custom protocol called "asset://". However, it is only available in remoteFiles definition right now.

Example:

 location: 'asset://file.js' points to the assets/file.js file.
 (equivalent to the io/viewer/get-plugin-asset/<your plugin id>/file.js)

 location: 'asset://images/logo.png' points to the assets/images/logo.png file.
 (equivalent to the io/viewer/get-plugin-asset/<your plugin id>/images/logo.png)

sdk commands

The SDK provides some commands accessible through Grunt.

Start the development server

To start the dev server and have it serve your plugin, run the following command:

$ grunt server

This will start a web server on port 9000 and open a web page in your default browser which will show the plugin simulator page.

Deploy your plugin

Run the following command to deploy your plugin to a (BIMAD) server.

$ grunt deploy

You will be prompted to enter a server url as well as a username and password. Users must have administrator roles, otherwise the deployment will fail.

This task will create ".xmp" file. This file is a new format for plugins which includes your plugin code and assets.

Undeploy your plugin

Run the following command to undeploy your plugin from the server.

$ grunt undeploy

You will be prompted to enter a server url as well as a username and password. Users must have administrator roles, otherwise the un-deployment will fail.

Create plugin archive

If you want to pass plugin to share with your co-worker, you can create archive file by running following command.

$ grunt archive

This command will create .xmp file in the sdk root directory. "grunt deploy" command also calls this before deploying.
